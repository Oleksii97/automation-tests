# 📋 Cypress Pagination Tests

Автоматизовані end-to-end тести пагінації DataTables на базі **Cypress**.  
Покриває повний цикл навігації: стартовий стан → прохід вперед/назад → прямі переходи → цілісність даних.

## ⚙️ Кастомні команди (`commands.js`)

Замість того щоб дублювати логіку в кожному тесті — всі повторювані дії винесені в кастомні команди. Зміна селектора = зміна в одному місці.

| Команда | Що робить |
|---|---|
| `cy.assertPage(n)` | Перевіряє що активна сторінка в пагінаторі = `n` |
| `cy.assertItemsCount(n)` | Перевіряє що в таблиці рівно `n` видимих рядків |
| `cy.assertDisabled(dtIdx)` | Перевіряє що кнопка пагінації неактивна (`disabled`) |
| `cy.assertEnabled(dtIdx)` | Перевіряє що кнопка пагінації активна (не `disabled`) |
| `cy.clickPage(dtIdx)` | Безпечний клік: спочатку перевіряє `not.disabled`, потім клікає |
| `cy.clickPageNumber(n)` | Клік на конкретну цифру сторінки в пагінаторі |

> **Чому `clickPage` а не просто `.click()`?**  
> Звичайний `.click()` на `disabled` кнопку не кине помилку одразу — тест «пройде» і впаде пізніше з незрозумілою причиною. `cy.clickPage` перевіряє стан кнопки **до** кліку і дає чітке повідомлення про помилку в потрібному місці.

## 🧪 Структура тестів (`pagination.cy.js`)

**Target URL:** `https://practice.expandtesting.com/dynamic-pagination-table`  
**Page size:** `3` рядки → `4` сторінки (12 записів / 3 = 4)

### Конфігурація

```js
const CONFIG = {
  url:      'https://practice.expandtesting.com/dynamic-pagination-table',
  pageSize: 3,
};
```

Усі «магічні числа» в одному місці — якщо сайт зміниться, достатньо оновити `CONFIG`.

---

### 🔵 Context 1 — Стартовий стан

Перевіряє що після завантаження сторінки всі елементи в очікуваному початковому стані.

| Тест | Що перевіряється |
|---|---|
| `відкривається перша сторінка` | Активна сторінка = 1 |
| `на першій сторінці рівно 3 рядки` | `assertItemsCount(3)` |
| `кнопка "Назад" задізейблена на сторінці 1` | `assertDisabled('previous')` |
| `кнопка "Вперед" активна на сторінці 1` | `assertEnabled('next')` |

---

### 🔵 Context 2 — Послідовний прохід вперед (1 → 4)

Перевіряє навігацію кнопкою «Next» крок за кроком до останньої сторінки.

| Тест | Сценарій |
|---|---|
| `перехід 1 → 2` | clickPage(next) → assertPage(2) |
| `перехід 1 → 2 → 3` | ×2 кліки → assertPage(3) |
| `перехід 1 → 2 → 3 → 4` | ×3 кліки → assertPage(4) |
| `кнопка "Вперед" задізейблена на сторінці 4` | Остання сторінка — Next disabled |
| `кнопка "Назад" активна на сторінці 4` | Остання сторінка — Prev enabled |

---

### 🔵 Context 3 — Послідовний прохід назад (4 → 1)

Вкладений `beforeEach` переводить кожен тест на сторінку 4 перед запуском.

| Тест | Сценарій |
|---|---|
| `перехід 4 → 3` | clickPage(prev) → assertPage(3) |
| `перехід 4 → 3 → 2` | ×2 кліки → assertPage(2) |
| `перехід 4 → 3 → 2 → 1` | ×3 кліки → assertPage(1) + assertDisabled(prev) |

---

### 🔵 Context 4 — Прямий перехід (кліки на номер сторінки)

Тести генеруються динамічно через `forEach` — не дублюємо один і той самий шаблон вручну.

```js
[1, 2, 3, 4].forEach((pageNum) => {
  it(`клік на сторінку ${pageNum}`, () => {
    cy.clickPageNumber(pageNum);
    cy.assertPage(pageNum);
  });
});
```

---

### 🔵 Context 5 — Крайні сторінки (Edge Cases)

| Тест | Сценарій |
|---|---|
| `сторінка 4 — Next disabled, Prev enabled` | Обидва стани перевіряються разом |
| `сторінка 1 — Prev disabled, Next enabled` | 4 → 1 → перевірка стану |
| `з сторінки 2 перейти одразу на 4` | Прямий перехід без проміжних кроків |

---

### 🔵 Context 6 — Цілісність даних

| Тест | Логіка |
|---|---|
| `немає дублювання рядків між сторінками` | Збираємо імена з усіх 4 сторінок → `Set` має той самий розмір що й масив |
| `загальна кількість рядків збігається з "Showing X to Y of Z"` | `total` по сторінках = число після `of` в DataTables info |

## 🔑 Ключові технічні рішення

### Чому `.within()` в `assertPage`?

На сторінці є breadcrumb з класом `.active` на `<li>`. Без `#example_paginate` Cypress міг знаходити breadcrumb замість пагінатора. `.within()` обмежує пошук тільки контейнером DataTables.

```js
cy.get('#example_paginate').within(() => {
  cy.get('.paginate_button.current, .paginate_button.active').first()...
});
```

### Чому `assertDisabled` перевіряє `<li>`, а не `<a>`?

DataTables ставить клас `disabled` на батьківський `<li>`, а не на `<a>`. Через це `should('be.disabled')` на `<a>` ніколи не спрацює — це не HTML-атрибут `disabled`, а CSS-клас на іншому елементі.

```js
cy.get(`[data-dt-idx="${dtIdx}"]`)
  .closest('li')
  .should('have.class', 'disabled');
```

### Чому `:visible` в `assertItemsCount`?

DataTables може ховати рядки через `display: none` замість видалення з DOM. Без `:visible` лічильник повертав би неправильне число.

```js
cy.get('#example tbody tr:visible').should('have.length', count);
```

## 📐 Принципи які закладені в код

- **DRY** — жодного дубльованого коду: кастомні команди, `CONFIG`, `forEach`-генерація тестів
- **Fail Fast** — `clickPage` перевіряє стан кнопки до кліку, щоб помилка виникала в правильному місці
- **Explicit Waiting** — явне очікування (`should('have.length', n)`) замість `cy.wait(ms)`
- **Isolation** — кожен тест стартує з чистого стану через `beforeEach`
- **Readable** — назви тестів читаються як речення: *«кнопка "Назад" задізейблена на сторінці 1»*
