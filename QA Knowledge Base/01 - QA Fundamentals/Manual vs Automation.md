# Manual vs Automation Testing

> Коли автоматизувати, а коли залишати ручне тестування.

#qa #automation #fundamentals #strategy

---

## Overview

Автоматизація — це інвестиція. Вона потребує часу на написання та підтримку. Не все варто автоматизувати.

---

## Comparison Table

| Критерій | Manual Testing | Automation Testing |
|----------|---------------|-------------------|
| **Швидкість виконання** | Повільно | Дуже швидко |
| **Вартість запуску** | Дорого (час людини) | Дешево (сервер) |
| **Вартість створення** | Дешево | Дорого (розробка) |
| **Стабільність** | Людська помилка | Детерміновано |
| **Підтримка** | Мінімальна | Потрібна при змінах |
| **Exploratory** | Відмінно | Погано |
| **Usability** | Відмінно | Неможливо |
| **Regression** | Повільно і нудно | Ідеально |
| **Нові фічі** | Краще спочатку | Після стабілізації |

## What to Automate

### Автоматизувати ВАРТО:

- ✅ Regression тести — запускаються часто
- ✅ Smoke тести — швидка перевірка після деплою
- ✅ API тести — стабільні, не залежать від UI
- ✅ Тести з великим обсягом даних — data-driven
- ✅ Тести, які часто повторюються
- ✅ Тести для різних браузерів/платформ
- ✅ Performance / Load тести
- ✅ Прекондиції для інших тестів (логін через API)

### Автоматизувати НЕ ВАРТО:

- ❌ Exploratory testing — вимагає людської інтуїції
- ❌ Usability / UX тести
- ❌ Тести, які змінюються дуже часто
- ❌ One-time тести — не окупляться
- ❌ Тести, які ще в процесі дизайну
- ❌ Captcha / WYSIWYG редактори
- ❌ Нові функції на стадії розробки

## ROI of Automation

Формула окупності:

```
ROI = (Manual Cost × Times Run) - Automation Cost
      ────────────────────────────────────────────
                    Automation Cost
```

**Приклад:**
- Ручний тест: 30 хв = $15 (за раз)
- Автоматизація: 4 години = $200 (одноразово)
- Запускається: 3 рази/тиждень × 52 тижні = 156 разів
- Ручна вартість за рік: 156 × $15 = $2,340
- **ROI = ($2,340 - $200) / $200 = 1070%** ← варто автоматизувати

## Automation Pyramid Strategy

```
Manual → Automation перехід:

1. Почни з API тестів (стабільні, швидкі)
2. Додай UI smoke тести (критичний шлях)
3. Розширюй regression suite
4. Performance тести
5. Security тести в CI/CD
```

## Common Mistakes

| Помилка | Чому погано | Рішення |
|---------|------------|---------|
| Автоматизувати все | Дорого підтримувати | Пріоритизуй за ROI |
| Починати з E2E | Повільно, нестабільно | Починай з API/Unit |
| Не оновлювати тести | Тести "гниють" | Code review для тестів |
| Ігнорувати flaky tests | Шум у CI | Аналізувати та виправляти |
| Не використовувати CI | Тести тільки локально | [[CI-CD Overview]] |
